\vspace{10pt}
{\large I built Second Opinion to get hands-on experience with LLM APIs and real-time streaming, using a domain I'm genuinely interested in as the testing ground: medicine. Most AI tools in this space just return a diagnosis. I wanted to see if a model could show its work.}

% ── What it does ────────────────────────────────────────────────────────────
\section{What It Does}

You describe a patient in free text. The model streams a live differential as it reasons: hypotheses appear, probability estimates shift, and supporting or refuting evidence attaches to each diagnosis in real time. Every step is visible as it happens, not just the conclusion.

After finishing, the model surfaces 2-3 clarifying questions - the ones that would most change the differential. You can answer them or add new clinical findings, and the model reruns with the full updated context. Each refinement pass is shown as a labeled block below the original presentation so you can see exactly what changed and why.

In compare mode, two models reason through the same case in parallel. Same patient, same moment, different reasoning paths. Where they agree the differential is more trustworthy. Where they disagree, that's often the most interesting part. Every run can be exported as a PDF and saved to session history.

% ── How it works ────────────────────────────────────────────────────────────
\section{How It Works}

The system prompt tells the model to respond only in NDJSON - one typed JSON object per line, nothing else. Seven event types are defined: \texttt{thought}, \texttt{hypothesis}, \texttt{update}, \texttt{evidence}, \texttt{tests}, \texttt{conclusion}, and \texttt{question}. The server streams these to the client over Server-Sent Events and the frontend parses each line as it arrives. Malformed lines are silently dropped server-side.

The prompt acts as a schema enforcer. The model isn't asked to ``think about'' a diagnosis - it's given a strict output contract with required field names, ordering rules, and cardinality constraints. That's what turns an open-ended language model into something that behaves like a structured reasoning engine.

% ── What's next ─────────────────────────────────────────────────────────────
\section{What's Next}

The part I'm most interested in is image analysis. Connecting visual reasoning over CT and MRI scans to the same differential pipeline - so the model can weigh radiological findings alongside the clinical presentation - feels like a natural next step. EHR integration is another obvious one, letting the model pull from real patient records rather than free text.
